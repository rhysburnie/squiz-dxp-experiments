# Squiz Astro Adapter — Build Spec

## Overview

A custom Astro integration + adapter that compiles `.astro` component files into
Squiz Component Service modules (`main.js`), while preserving all framework CSS
and JS assets (Vue, React, Svelte, etc.) as deduplicated static bundles that
Squiz can reference as global or per-component asset payloads.

---

## Goals

- Author Squiz components as `.astro` files
- SSR each component to a flat HTML string (no Shadow DOM, no DSD)
- Output a `main.js` per component matching the Squiz render function contract
- Collect all framework/component CSS and JS into a shared asset bundle
- Allow third parties to drop in React/Svelte/other islands alongside Vue base components
- No framework lock-in at the component consumer level — partners only interact with Squiz

---

## Directory Structure

```
packages/
  squiz-astro-adapter/       # the custom Astro adapter/integration package
  components/                # shared Vue (and other) base components
    src/
      Button/
      Card/
      ...

squiz-components/            # one dir per Squiz component
  src/
    hero-banner/
      hero-banner.astro      # Squiz component definition
      hero-banner.types.ts   # prop schema matching Squiz data model
    article-card/
      article-card.astro
      article-card.types.ts

dist/                        # build output (consumed by Squiz CLI deploy)
  hero-banner/
    main.js                  # Squiz render module
  article-card/
    main.js
  _assets/
    bundle.js                # deduplicated client JS (Vue runtime, etc.)
    bundle.css               # deduplicated CSS from all component imports
```

## Squiz `main.js` Contract

Each Squiz component module must export a default function that accepts a props
object (JSON from the CMS data model) and returns an HTML string.

```js
// dist/hero-banner/main.js (target output shape)
export default function render(props) {
  return `<section class="hero-banner">...</section>`
}
```

Props are passed from Squiz's data model at request time. The render function
must be synchronous from Squiz's perspective — async resolution happens during
the build/pre-render step, not at serve time.

## Authoring Format

Each Squiz component is a `.astro` file. The frontmatter receives the Squiz
props object, passes it to component imports, and the template renders the
output HTML.

```astro
---
// hero-banner.astro
import HeroBanner from '@components/HeroBanner/HeroBanner.vue'

// Props injected from Squiz data model via adapter at build time
const { heading, subheading, imageUrl, ctaLabel, ctaUrl } = Astro.props
---

<HeroBanner
  heading={heading}
  subheading={subheading}
  imageUrl={imageUrl}
  ctaLabel={ctaLabel}
  ctaUrl={ctaUrl}
/>
```

Third-party React islands follow the same pattern using `client:load` or
`client:idle` directives where interactivity is required:

```astro
---
import HeroBanner from '@components/HeroBanner/HeroBanner.vue'
import PartnerWidget from '@partner/widget/Widget.jsx'
---

<HeroBanner heading={Astro.props.heading} />
<PartnerWidget client:idle data={Astro.props.widgetData} />
```

## Adapter Behaviour

### Build Phase (`astro build`)

1. Astro builds all `.astro` component files using `output: 'static'`
2. Each component route renders to a static HTML string
3. The adapter's `astro:build:generated` hook intercepts all generated routes
4. For each route (component), the adapter:
   - Reads the rendered HTML string
   - Wraps it in the Squiz render function signature
   - Writes `dist/{component-name}/main.js`
5. All JS and CSS assets collected from framework imports (Vue runtime, component
   styles, React runtime if used) are:
   - Deduplicated across all components (Vite handles this naturally via chunking)
   - Written to `dist/_assets/bundle.js` and `dist/_assets/bundle.css`
   - Referenced in the Squiz parse file global template as static includes

### Asset Collection Strategy

Vite's default chunking already deduplicates shared modules (e.g. the Vue
runtime will appear once across all components). The adapter reads the Vite
manifest (`dist/_astro/manifest.json`) after build to:

- Identify all JS chunks and CSS files
- Separate framework/component assets from Astro's own runtime glue
- Output a clean asset manifest for use by the Squiz deploy step

### Props Passthrough

At Squiz serve time, the component's `main.js` receives the CMS data model as
a plain JS object. Because the HTML is pre-rendered at build time, the render
function for purely static components simply returns the pre-baked string:

```js
// Static component — HTML baked at build time
export default function render(props) {
  return PRERENDERED_HTML  // injected as a template literal at build time
}
```

For components with dynamic content (i.e. content that varies per CMS entry),
the build step generates a template function instead:

```js
// Dynamic component — props interpolated at serve time
export default function render(props) {
  return `<section class="hero">
    <h1>${props.heading}</h1>
    <p>${props.subheading}</p>
  </section>`
}
```

The adapter distinguishes static vs dynamic by the presence of Squiz prop
references in the `.astro` frontmatter — components that destructure
`Astro.props` are treated as dynamic.

## Open Questions

1. **Prop schema validation** — Should `hero-banner.types.ts` be used to
   validate Squiz's incoming data model at build time, or only as a dev-time
   contract? Consider using Zod for runtime validation in dev.

2. **Client hydration budget** — For components with `client:*` islands, the
   JS bundle grows. Squiz's deduplication feature handles repeated includes, but
   we need to confirm the parse file setup handles this correctly for the
   `_assets/bundle.js` approach.

3. **Incremental builds** — Squiz deploys per-component via CLI. Should the
   adapter support building a single component in isolation (e.g.
   `astro build --component hero-banner`) to avoid full rebuilds on small changes?

4. **Third-party onboarding** — Partners need a template repo. The authoring
   surface they see should be just the `.astro` file and the types file.
   All adapter wiring should be invisible to them.

5. **CSS scoping** — Vue SFC `<style scoped>` and Astro's own scoped styles both
   emit hashed class names. These need to survive the HTML string extraction and
   remain consistent between the pre-rendered HTML and the emitted CSS bundle.

---

## Non-Goals (v1)

- No runtime SSR server (all pre-rendered at build time)
- No Shadow DOM / Declarative Shadow DOM output
- No Squiz CMS Content API integration (data is passed in as props by Squiz at
  serve time, not fetched by the component)
- No support for Squiz inline editing hooks (v2 consideration)
